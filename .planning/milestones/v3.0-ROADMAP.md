# Milestone v3.0 — Refinement & Optimization

## Scope
Code refinement pass: fix bugs, eliminate dead code, refactor for clarity, and optimize performance across the entire RedWardenLite codebase.

## Phases

### Phase 11: Bug Fixes & Dead Code Removal
- Fix identified bugs (bare excepts, unreachable code, reference errors)
- Remove dead/unreachable code paths
- Fix typos in user-facing strings and log messages

### Phase 12: Refactoring & Code Quality
- Extract the massive `_my_handle_request` method into focused helpers
- Reduce `redirector.py` complexity by moving validation logic to policy check modules
- Consolidate duplicated encoding/decoding logic
- Improve type safety and defensive patterns

### Phase 13: Performance & Optimization
- Optimize hot paths (header processing, body encoding/decoding)
- Reduce unnecessary object allocations
- Improve async efficiency
- Add `__slots__` where beneficial

## Identified Issues

### Bugs
1. `proxyhandler.py:generate_ssl_certificate()` — uses `self.logger` in a `@staticmethod` (will crash)
2. `proxyhandler.py:_my_handle_request()` — references `self.client_address` which doesn't exist on Tornado handler
3. `proxyhandler.py:_run_shadow_parity()` — missing `verify_upstream_tls` kwarg in shadow `_fetch_upstream` call
4. `proxyhandler.py:my_handle_request()` — references `self.request.client_address` before it's set
5. `proxyhandler.py:connect_relay()` — references `self.close_connection` which doesn't exist on Tornado
6. `optionsparser.py:parseParametersFromConfigFile()` — uses `self.logger` but it's a module-level function (no self)
7. `RedWardenLite.py:main()` — `type(port) == int` check but ports are strings from argparse; int ports skip `:`/`/` parsing
8. `redirector.py:_drop_request_without_expected_header_value_check()` — `req.headers[k.lower()]` will KeyError if header missing
9. `proxyhandler.py:_my_handle_request()` — `options['allow_invalid']` referenced but never set (should be `allow_invalid` from argparse)

### Dead Code / Unreachable
1. `proxyhandler.py` — `return False` after `raise` in `main()` exception handlers (unreachable)
2. `utils.py` — `send_request_decorator` and `putheader_decorator` monkeypatch `http.client` but proxy uses `requests` library
3. `proxyhandler.py:filter_headers()` — defined but never called (commented out at call sites)
4. `proxyhandler.py:log_message()` / `log_error()` — reference `self.address_string()` and `self.log_date_time_string()` which don't exist on Tornado handler

### Refactoring Opportunities
1. `_my_handle_request()` is ~400 lines — break into fetch/response/logging phases
2. `redirector.py:help()` is ~250 lines doing config parsing — should be a separate init method
3. Duplicated body encode/decode exists in both `proxyhandler.py` and `transport_parity.py`
4. `my_handle_request()` access log formatting is ~80 lines that should be a separate function
5. Multiple `type(x) == str` checks should use `isinstance()`
6. Bare `except:` blocks throughout should catch specific exceptions

### Performance
1. `_perform_requests_fetch()` uses synchronous `requests.request()` even in async mode (wrapped in `to_thread`) — could use `tornado.httpclient.AsyncHTTPClient` natively
2. `ProxyRequestHandler.get_ip()` does DNS resolution on every handler construction
3. `observability.py:should_emit_request_event()` computes SHA256 on every request even at rate=1.0
4. Global `alreadyPrintedPeers` set in redirector.py grows unbounded
